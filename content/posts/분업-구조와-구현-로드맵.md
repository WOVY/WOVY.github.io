지난 글에서 bokjidream의 전체 구조를 소개했습니다. 이번 글에서는 제가 담당하는 AI 파트(`ai/`)를 두 명이 어떻게 나눠 개발하는지, 그리고 Phase별로 무엇을 만들어 나가는지를 정리합니다.

## 분업 원칙

개발 인원은 2명(A, B)입니다. 역할을 나누기 전에 한 가지를 먼저 합의했습니다.

> **Phase 1은 공동 작업.**

LangGraph의 `StateGraph`, `Checkpointer`, `interrupt()` 패턴은 이 프로젝트에서 가장 어려운 구간입니다. 혼자 짜고 넘기면 나머지 한 명이 맥락 없이 그 위에서 작업해야 합니다. 둘이 함께 구현하고 학습한 뒤, Phase 2부터 병렬로 나눕니다.

Phase 2 이후 분업 기준은 노드의 핵심 패턴입니다.

- **A** — LLM과 대화하는 인터뷰 계열 노드 + 공통 도구 인프라 + 출력 노드
- **B** — RAG HTTP 연동 계열 노드 + HitL(사람이 개입하는) 선택 노드 + 서버

두 패턴이 완전히 다르기 때문에 서로 간섭 없이 병렬로 작업할 수 있습니다.

## 노드별 소유권

8개 노드를 이렇게 나눴습니다.

| 노드 | 담당 | 핵심 패턴 |
| --- | --- | --- |
| `initial_interview` | A | LLM 대화 → structured output → 누락 필드 감지 → 재진입 루프 |
| `rag_search` | B | JSON 전송 → RAG HTTP 호출 → 결과 파싱 |
| `service_select` | B | `interrupt()` HitL — 사용자가 서비스 목록에서 직접 선택 |
| `rag_detail` | B | RAG HTTP 상세 조회 → LLM으로 누락 필드 추론 |
| `detail_interview` | A | 서비스 특화 2단계 인터뷰, `extra:` 접두사 필드 처리 |
| `document_guidance` | A | 사용자 상황 기반 서류 목록 필터링 |
| `draft_writer` | B | 신청서 항목별 작성 가이드 생성 |
| `report_writer` | A | 전체 결과를 Markdown으로 재구성 |

`graph/state.py`와 `graph/builder.py`는 공동 작업입니다.

## Phase별 로드맵

### Phase 1 — LangGraph 기반 구축 (공동)

가장 먼저 해야 할 일은 그래프의 뼈대를 세우는 것입니다.

- `graph/state.py` : `UserProfile`, `WelfareCandidate`, `AgentState` Pydantic 모델 정의
- `graph/builder.py` : StateGraph 노드 등록·엣지 연결·조건부 엣지·Checkpointer 설정

완료 기준은 그래프가 START → END까지 컴파일되고, `interrupt()` → resume 흐름이 확인되는 것입니다. 이 뼈대 위에 각자 노드를 얹는 구조이기 때문에 여기서 흔들리면 이후가 전부 흔들립니다.

### Phase 2 — 에이전트 노드 구현 (A·B 병렬)

Phase 1이 끝나면 A와 B가 동시에 작업을 시작합니다. 단, 한 가지 선행 조건이 있습니다.

> **RAG API 계약을 먼저 확정한다.**

B가 짜는 `rag_search`, `rag_detail` 노드는 RAG 서비스(`rag/`)의 응답 형식에 의존합니다. 계약 없이 시작하면 나중에 인터페이스가 어긋납니다. Phase 2 시작 전 `rag/` 팀과 JSON 스키마를 확정하고 시작합니다.

**A의 작업 순서**

```plain
tools/ (llm.py, prompt_loader.py)
  → initial_interview
  → detail_interview
  → document_guidance
  → report_writer
```

**B의 작업 순서**

```plain
tools/rag_client.py (스텁 구현)
  → rag_search
  → rag_detail
  → service_select
  → draft_writer
```

B는 실제 RAG 서비스가 준비되기 전까지 `rag_client.py`에 더미 스텁을 구현해 인터페이스 계약만 맞춰둡니다. Phase 3에서 실제 HTTP 구현으로 교체합니다.

### Phase 3 — RAG 실제 연동 (B)

`rag_client.py`의 스텁을 실제 API에 맞게 교체합니다.

### Phase 4 — 로컬 LLM 전환 (A)

개발 초기에는 편의를 위해 Google Gemini를 쓰고 있습니다. `tools/llm.py`의 `get_llm()` 함수 한 곳만 바꾸면 Ollama 기반 Llama 3으로 전환됩니다.

### Phase 5 — 최종 통합 (A·B 병렬)

- A : `main.py` 대화형 CLI 완성
- B : `server.py` FastAPI 서버 (`POST /chat`, `POST /resume`)
- A·B : E2E 시나리오 테스트 3개, README 업데이트

## 설계에서 신경 쓴 것들

**RAG 결과 없음 처리.** RAG에서 결과를 찾지 못하면 LLM이 임의로 복지 서비스를 만들어내지 않도록 설계했습니다. 결과가 없으면 "해당 서비스를 찾을 수 없음"을 안내하고 파이프라인을 종료합니다. LLM으로 넘겨 편의성을 챙기기보다는 정확성이 더 중요하기 때문입니다.

**HitL(Human-in-the-Loop).** `service_select` 노드에서 사용자가 직접 서비스 목록을 보고 선택합니다. LangGraph의 `interrupt()`로 그래프 실행을 멈추고, 사용자 입력을 받은 뒤 재개하는 방식입니다. 

**extra_fields.** `UserProfile`에는 미리 정의된 필드 외에 `extra_fields: dict`를 뒀습니다. RAG 상세 정보에서 요구하지만 모델에 아직 없는 항목(예: `deposit_amount`, `vehicle_owned`)을 동적으로 저장하기 위해 정의했습니다. 추후 자주 나오는 필드가 있다면 수정할 예정입니다.
